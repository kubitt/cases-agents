# Dunning-последовательность

Цель: каждое касание — в нужный день, нужным тоном, с точными реквизитами. Тон растёт со стадией, уважение — константа.

## Стадии (дефолт «стандарт»; интервалы и профили — в DUNNING_PLAYBOOK.md)
| Стадия | Когда | Тон |
|---|---|---|
| S0 | за 3 дня до due_date | сервисное напоминание: «всё ли есть для оплаты» |
| S1 | 1–3 дня просрочки | вежливое «возможно, упустили» |
| S2 | 7–10 дней | деловое: сумма, срок, просьба назвать дату оплаты |
| S3 | 14–21 день | твёрдое: история напоминаний, конкретный дедлайн, последствия по правилу пользователя (стоп-услуги) |
| S4 | 30+ дней | финальное: фиксация фактов, запрос решения; параллельно эскалация пользователю |

## Алгоритм
1. Возьми из реестра: invoice_id, client, contact_name, email, amount, currency, due_date, days_overdue, stage, status, promise_date, vip, last_touch, next_touch.
2. Пропусти счёт, если: status ∈ {paid, paid_claimed, disputed, escalated, written_off}; или promised и promise_date ≥ сегодня; или vip=да без прямой команды пользователя; или next_touch в будущем.
3. Определи стадию по days_overdue (таблица). Стадию вниз не откатывай: было S2, а по дням выходит S1 — оставь S2.
4. Возьми шаблон стадии из DUNNING_PLAYBOOK.md, подставь: имя контакта, номер счёта, сумму с валютой, срок; для S2+ — даты прошлых напоминаний. Несколько счетов одного клиента — одно письмо с таблицей счетов, стадия по самому старому.
5. Черновик в Gmail: ответом в существующий тред клиента, если он есть (видна история), иначе новое письмо.
6. После отправки пользователем обнови реестр: stage, last_touch=сегодня, next_touch=сегодня+интервал стадии, запись в thread_log.

## Шаблон выхода (сводка пользователю)
```
Черновики (3):
1. ООО Вектор — счёт 214 — 84 000 ₽ — S2 (9 дн) — просим дату оплаты
2. ИП Крылов — счета 198, 201 — 36 500 ₽ — S3 (16 дн) — объединённое письмо
3. Студия Норд — счёт 220 — 120 000 ₽ — S0 (срок 09.07) — напоминание до срока
Утвердить: все / номера. Пропустить: «пропусти N».
```

## Краевые случаи
- Нет email у клиента → строка «нет контакта» в сводке, письмо не готовить.
- Частичная оплата (подтвердил пользователь) → в письме остаток, в реестре amount=остаток и пометка partial в notes.
- Обещанная дата уже прошла к моменту прохода → письмо-возврат к обещанию (шаблон в плейбуке), не очередная стадия.
- days_overdue > 90 → новые письма не готовить автоматически: в сводке блок «Эскалация» с вариантами (личный контакт, юрист, списание) — решение пользователя.
