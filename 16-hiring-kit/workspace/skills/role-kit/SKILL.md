# Кит по роли

Цель: согласованный комплект — из JD следуют вопросы, из вопросов scorecards; нанимать можно сразу.

## Алгоритм
1. Профиль роли — 4 вопроса пользователю (если ответы не даны заранее): зачем роль появилась; что человек должен СДЕЛАТЬ за 90 дней (3 результата); кому подчиняется и с кем работает; вилка-ощущение (для сверки с бенчмарком).
2. JD (в Google Docs, структура — knowledge/KIT_TEMPLATE.md): о компании (из COMPANY_PROFILE.md, без штампов) → миссия роли одной фразой → результаты 90 дней → задачи → must-have (≤5, проверяемые) → nice-to-have → условия и вилка (после бенчмарка) → как откликнуться.
3. Воронка 4 этапа: скрининг 20 мин (мотивация+факты) → профессиональное 60 мин (глубина по must-have) → кейс/тестовое (реальная мини-задача роли; оплату тестового >2ч — рекомендуй) → финал 30 мин (ценности, ожидания, оффер-разговор). На каждый этап: цель, кто проводит, порог прохождения.
4. Банк вопросов: на каждый must-have — 2–3 поведенческих вопроса («расскажите про случай, когда…») + «что слушать в ответе» (маркеры сильного/слабого ответа). Без вопросов-загадок и «кем вы видите себя через 5 лет».
5. Scorecards: таблица на этап — критерий (из JD) / вопросы-источники / шкала 1–4 с описанием уровней / вес. Итог: правило решения (например, «двойка по любому must-have = нет»).
6. Постинги из JD: hh-версия (полная), telegram-версия (≤800 знаков, живым языком, вилка и 3 главных условия), LinkedIn-стиль при необходимости.
7. Сохрани папкой «Кит: <роль>» в Docs; строку роли — в Sheets «Реестр найма».

## Краевые случаи
- Пользователь хочет «многорукого» (маркетолог+дизайнер+SMM в одном) → покажи вилки каждой роли и риск, предложи главную роль + подрядчиков.
- Роль-новинка без рыночных аналогов → собери из ближайших ролей, пометь допущения.
- Экспорт для скринера: по команде — рубрика в формате must-have/nice-to-have с весами, файлом в Docs/Sheets.
